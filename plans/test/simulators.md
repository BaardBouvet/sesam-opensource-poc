# Simulated Source Systems

Build fake HubSpot and Tripletex APIs so we can develop and test without real accounts. Must support configurable jitter and failure injection.

Webhook behavior is required in simulators (not optional) so webhook receiver and orchestration paths can be tested end-to-end.

Phase 1 simulator profile:
- Seed both correlated and uncorrelated records to observe linkage behavior.
- Default seed size is ~10 records per main entity, with configurable scale for performance tests.
- Simulator must run standalone (outside test harness) for demos and manual validation.
- Simulator must provide a Web UI with table views and live updates when data changes.

## Implementation Strategy

### Option A: Hardcoded behavior only

- Fastest to deliver for limited entity scope.
- Simple to debug and reason about in Phase 1.
- Becomes harder to evolve as simulated API surface grows.

### Option B: Metadata-driven behavior only

- Better long-term extensibility and reuse.
- Enables declarative simulator behavior/profile definitions.
- Higher upfront complexity and slower initial delivery.

### Option C: Hybrid (Recommended)

- Phase 1: hardcoded endpoint behavior + configurable seed/traffic/error settings.
- Phase 2: introduce metadata-driven controls incrementally (field profiles, webhook registration templates, jitter/error policies).

Recommendation:
- Use **Hybrid** with hardcoded-first delivery in Phase 1.

### Option A: FastAPI Mock Servers — Recommended

- **What:** Build lightweight Python FastAPI apps that implement the subset of HubSpot/Tripletex APIs we use. Backed by PostgreSQL.
- **Pros:**
  - Same language as the rest of the stack
  - Full control over behavior, jitter, and failure modes
  - Can generate realistic test data with Faker
  - Easy to containerize and use in testcontainers
  - OpenAPI spec auto-generated by FastAPI
- **Cons:**
  - Must manually implement each endpoint we use
  - Risk of simulator drift from real API behavior
- **Features:**
  - Configurable response latency (jitter)
  - Configurable error rates (429 rate limit, 500 server error, timeout)
  - Seed data generation with mixed correlated/uncorrelated records
  - Default seed profile around 10 records per entity, configurable via admin/UI settings
  - State: changes persist in memory during test run
  - Webhook emit support for create/update/delete changes
  - Webhook registration support (create/list/update/delete subscriptions)
  - Subscription status lifecycle simulation (active, disabled, error states)
  - Standalone mode (run as local demo app without test harness)
  - Built-in Web UI showing source tables and webhook/subscription state with live updates
  - Admin endpoints: `POST /admin/reset`, `POST /admin/configure-jitter`, `POST /admin/emit-webhook`
- **Effort:** Medium

### Option B: WireMock / Prism

- **What:** Use API mocking tools that can replay recorded responses or generate from OpenAPI specs.
- **Pros:**
  - Can record real API traffic and replay it
  - Prism can generate mocks directly from OpenAPI specs
  - No custom code for basic mocking
- **Cons:**
  - Limited stateful behavior (WireMock stateful scenarios are clunky)
  - Harder to implement custom jitter/failure injection
  - Tripletex may not have a public OpenAPI spec
  - Java-based (WireMock) — different from rest of stack
- **Effort:** Low for basic mocking, High for stateful + failure injection

### Option C: Testcontainers + Recorded Fixtures

- **What:** Use pre-recorded JSON fixtures loaded into a generic mock server at test time via testcontainers.
- **Pros:**
  - Very fast test startup
  - Deterministic — same data every time
- **Cons:**
  - No stateful behavior (can't test write-back)
  - Fixtures go stale
  - Not useful for development, only unit/integration tests
- **Effort:** Low

## Simulator Storage

### Option A: PostgreSQL (Recommended)

- Use the **same PostgreSQL instance** as the rest of the system, with strict isolation.
- Store simulator data in dedicated schemas (e.g., `sim_hubspot`, `sim_tripletex`, `sim_webhooks`).
- Enforce DB role separation so model/MDM readers cannot `SELECT` simulator schemas.
- Pros: realistic persistence behavior, closer parity with production data path, supports larger seed volumes/performance tests.
- Cons: requires explicit schema/permission management.

### Option B: SQLite / in-memory

- Pros: minimal setup and very fast startup.
- Cons: lower parity with target runtime behavior and weaker concurrency realism.

Recommendation:
- Use the same PostgreSQL instance in Phase 1 with schema + permission isolation.

Isolation requirements:
- Simulator writer role: read/write simulator schemas only.
- Model/MDM role: no read permissions to simulator schemas.
- Optional integration bridge (if needed): controlled views/materialization into staging schemas, not direct simulator schema reads.

## Simulator Scope

| Endpoint                        | HubSpot          | Tripletex        |
|---------------------------------|------------------|------------------|
| List contacts/customers         | GET /crm/v3/...  | GET /v2/...      |
| Get single contact/customer     | GET /crm/v3/...  | GET /v2/...      |
| Create contact/customer         | POST             | POST             |
| Update contact/customer         | PATCH            | PUT              |
| List companies                  | GET /crm/v3/...  | GET /v2/...      |
| Create/Update company           | POST/PATCH       | POST/PUT         |
| Search (if needed)              | POST /search     | GET with filters |
| Webhook callback emission       | yes (simulated)  | yes (simulated)  |
| Webhook registration            | yes (simulated)  | yes (simulated)  |

## Webhook Simulation Requirements

- Simulators must emit webhook callbacks on create/update/delete for entities in scope.
- Simulators must expose webhook registration APIs/state used by tests.
- Simulators must support subscription lifecycle transitions, including disabled/error states.
- Simulators must support replaying historical simulated events for receiver recovery tests.
- Simulators must support out-of-order and duplicate webhook delivery modes.
- Simulator webhook payloads should match provider-specific callback shapes used in this PoC.

Registration behavior by provider mode:
- HubSpot simulator: support explicit registration state and optional "manual mode" where registrations must be pre-seeded/configured.
- Tripletex simulator: support API-driven subscription management matching `GET/POST/PUT` subscription flows.

## Web UI Requirements

- Provide table-based views for core entities (contacts, companies/customers, associations).
- Provide webhook/subscription status views (including disabled/error states).
- Reflect data changes live (polling or push-based updates).
- Allow seed/config controls (record counts, correlated/uncorrelated mix, jitter/error profile).
- Include simple filters/search to inspect linkage-relevant records.

## Web UI Framework Options

### Option A: HTMX + Jinja2 templates (Recommended for Phase 1)

- Pros: fastest implementation on FastAPI stack, low frontend complexity, good fit for table-heavy simulator UI.
- Cons: less ergonomic for complex client-side interactions as scope grows.

### Option B: React + TypeScript (Vite)

- Pros: strong ecosystem and component/table tooling, scales well for richer UX.
- Cons: higher initial setup and maintenance cost.

### Option C: Vue 3 + TypeScript (Vite)

- Pros: concise component model, good DX, strong table/form ecosystem.
- Cons: smaller team familiarity/ecosystem in some environments.

Recommendation:
- Use **HTMX + Jinja2** for Phase 1 simulator UI.
- Reassess React/Vue if simulator UI evolves into a long-lived product surface.

## Live Update Strategy

### Option A: Polling (Recommended for Phase 1)

- Refresh table/status data on a short interval (e.g. 1-2s).
- Pros: simplest implementation, robust in dev/test environments, low operational complexity.
- Cons: extra repeated requests and slightly less real-time feel.

### Option B: Server-Sent Events (SSE)

- Push one-way updates from server to UI over long-lived HTTP stream.
- Pros: near real-time updates with lower overhead than polling for active screens.
- Cons: more connection/reconnect handling complexity than polling.

### Option C: WebSockets

- Full-duplex realtime channel between UI and simulator backend.
- Pros: best flexibility for high-frequency and bidirectional interactions.
- Cons: highest complexity (connection lifecycle, auth/session handling, testing complexity).

### Recommendation

- Use **Polling** in Phase 1 for fastest delivery and reliability.
- Re-evaluate for **SSE** in Phase 2 if update frequency/UX requirements exceed polling comfort.
